# BIG IDEA & CREATIVE CONCEPT

## Big Idea: ✅ **"จาก Gate Keeper สู่ Gate Opener" — "Thailand Opens for You"**

> **Core Concept**: BOI เว็บเดิม = Gate Keeper (กั้นนักลงทุน ด้วยข้อมูลหายาก ขั้นตอนซับซ้อน)
> BOI เว็บใหม่ = **Gate Opener** (เปิดประตูให้นักลงทุน — เปิดข้อมูล เปิดโอกาส เปิดทาง)
> Tagline บนเว็บ: **"Thailand Opens for You"**

### Transformation Story (Before → After)

| BEFORE — Gate Keeper | AFTER — Gate Opener |
|---|---|
| หาข้อมูลไม่เจอ ต้องจ้างที่ปรึกษา | ค้นหาเจอใน 3 คลิก |
| ไม่รู้ขั้นตอนยื่นขอ | Step-by-step Guide ชัดเจน |
| ติดตามสถานะไม่ได้ | ดูสถานะ Real-time 24/7 |
| ภาษาเดียว / ไม่ Mobile Friendly | 7 ภาษา / Mobile-First |
| เว็บช้า / ไม่ปลอดภัย | เร็ว ปลอดภัย ได้มาตรฐาน |

### Brand Gap Analysis (5 มิติ)

| มิติ | สถานะ | ปัญหา |
|---|---|---|
| Perception (การรับรู้) | สูง | นักลงทุนรู้จัก BOI ทั่วโลก |
| Personality (บุคลิก) | ทางการ | ภาพลักษณ์ราชการ จริงจัง |
| Image (ภาพลักษณ์) | **ล้าสมัย** | เว็บไม่ทันสมัย ไม่สะท้อนศักยภาพจริง |
| Value (คุณค่า) | สูงมาก | 60+ ปี ประวัติศาสตร์ ความน่าเชื่อถือ |

> **Key Insight**: Brand Value สูง (60+ ปี) แต่ Image ไม่สะท้อนคุณค่าจริง — **เว็บไซต์ต้องปิดช่องว่างนี้**

### ปัญหาเว็บไซต์ปัจจุบัน — 6 หมวด

| หมวด | ปัญหา |
|---|---|
| มาตรฐาน | ไม่ผ่าน ITA / DGA 3.0 ไม่ครบ / WCAG ไม่ผ่าน AA |
| ความปลอดภัย | ถูก Cyber Attack บ่อย / OWASP ไม่ครบ |
| UX/UI | ไม่ Responsive / Navigation ซับซ้อน / ไม่ Mobile-First |
| SEO | สำนักกฎหมาย rank สูงกว่า BOI / Meta ไม่ครบ |
| เทคนิค | PageSpeed ต่ำ / CMS ล้าสมัย / ไม่มี Cache Layer |
| Real-time | ข้อมูลไม่ update / ไม่เชื่อม API / สถานะดูไม่ได้ |

### Big Idea → Design Direction

| สิ่งที่ Big Idea กำหนด | รายละเอียด |
|---|---|
| **ทุกหน้า** | ต้อง "เปิด" บางอย่างให้ user (เปิดข้อมูล, เปิดขั้นตอน, เปิดโอกาส) |
| **Animation** | ประตูเปิด, เนื้อหาเผยออก, ทางเดินปรากฏ, แสงส่อง |
| **CTA** | "Open your..." / "เปิดประตูสู่..." / "Start your journey" |
| **Tone** | มั่นใจ เชิญชวน — ไม่ใช่ราชการขอข้อมูล |

### Re-framing ที่สำคัญ

| ความเข้าใจเดิม (ผิด) | ความเข้าใจใหม่ (ถูก) |
|---|---|
| BOI = หน่วยงานราชการ | BOI = **Gate Opener** |
| เว็บไซต์ = ให้ข้อมูล | เว็บไซต์ = **เปิดประตูสู่การลงทุน** |
| ผู้ใช้ = ต้องการค้นหา | ผู้ใช้ = ต้องถูก **invite + guide** |
| เป้าหมาย = ใช้งานง่าย | เป้าหมาย = **ตัดสินใจลงทุน** |

### Key Insight
> **นักลงทุนตัดสินใจด้วย HEAD (เหตุผล) แต่ commit ด้วย HEART (อารมณ์)**
> เว็บไซต์ต้อง satisfy HEAD + win HEART
> Big Idea "Gate Opener" ตอบทั้งสองด้าน: HEAD = ข้อมูลครบ ชัด เร็ว | HEART = ต้อนรับ เปิดกว้าง มั่นใจ

### Creative Pillars (3 เสาหลัก — สอดคล้องกับ Gate Opener)

| Pillar | Concept | WOW Factor | เชื่อมกับ Gate Opener |
|---|---|---|---|
| 🎯 Personalized | "Know Your Investment" | ระบบรู้จักผู้ใช้ แนะนำสิทธิประโยชน์ที่เหมาะสม | เปิดประตูที่ใช่ให้แต่ละคน |
| ⚡ Instant | "Answers in Seconds" | AI Chat + Smart Search ตอบได้ทุกคำถาม | เปิดคำตอบทันที ไม่ต้องรอ |
| 🔒 Trusted | "Security-First Design" | Zero Trust Architecture, PDPA Ready | เปิดด้วยความมั่นใจ ปลอดภัย |

### Slide 7: Project Methodology
**หัวข้อ**: "Double Diamond + Agile — วิธีการที่พิสูจน์แล้วว่าสำเร็จ"
**รายละเอียด**:
- Phase 1: Discover (UX Research) — เดือน 1-2
- Phase 2: Define (IA + Requirements) — เดือน 2-3
- Phase 3: Design (UI/UX + Prototype) — เดือน 3-5
- Phase 4: Develop (Next.js + CMS) — เดือน 4-8
- Phase 5: Test & Deploy — เดือน 8-10
- Phase 6: Training & Handover — เดือน 10-12
- แสดงว่า Design กับ Development ทำ parallel ได้

**Script**:
> "เราใช้ Double Diamond framework สำหรับ UX Research ผสม Agile Sprint สำหรับ Development จุดเด่นคือเราสามารถทำ Design และ Development คู่ขนานกันได้ ลด time-to-delivery โดยไม่เสียคุณภาพ"

**กรรมการอาจถาม**:
- "ทำ parallel ได้ยังไง จะไม่ conflict กันเหรอ?" → อธิบาย Design System approach: ออกแบบ component ก่อน → dev เริ่ม build foundation ระหว่าง design ทำ page layout
- "Sprint cycle เท่าไหร่?" → 2 สัปดาห์ต่อ Sprint, demo ทุก Sprint

### Slide 16: ทำไมต้อง Next.js (Deep Dive)
**หัวข้อ**: "Next.js — ไม่ใช่แค่ requirement แต่คือ best choice"

- Security: built-in XSS protection, CSRF protection
- เปรียบเทียบ: Next.js vs PHP เดิม → แสดง benchmark ความเร็ว

**Script**:
> "Next.js มี 3 โหมด rendering: SSR สำหรับหน้าที่เปลี่ยนบ่อย, SSG สำหรับหน้าที่ไม่ค่อยเปลี่ยน ทำให้โหลดภายใน 1 วินาที, และ ISR ที่ผสมข้อดีทั้งสอง อัปเดตเนื้อหาได้โดยไม่ต้อง rebuild ทั้ง site สำหรับเว็บ BOI ที่มีทั้งข่าว (เปลี่ยนทุกวัน) และข้อมูลนโยบาย (เปลี่ยนไม่บ่อย) Next.js เป็นคำตอบที่สมบูรณ์แบบ"

### Slide 26: Accessibility (WCAG 2.1 AA + DGA 3.0)

- Focus visible ทุก interactive element

- ทดสอบ: achecker.ca + Lighthouse >= 90 + W3C Validator 0 errors

### Slide 28: น้องอะไร — AI ที่เข้าใจ BOI
**หัวข้อ**: "Introducing 'Arai' (น้องอะไร) — Your BOI AI Assistant"
**รายละเอียด**:
- ชื่อ: **Arai (อะไร)** — ภาษาไทยแปลว่า "what?" / ต่างชาติออกเสียงง่าย / ตรงประเด็น = "ถามอะไรก็ตอบ"
- Engine: OpenRouter API → เชื่อมกับ LLM (Claude/GPT)
- Knowledge Control: **RAG (Retrieval-Augmented Generation)**
  - Vector Database เก็บข้อมูลจากเว็บ BOI ทั้งหมด
  - ตอบได้เฉพาะข้อมูลที่ train ไว้ + ข้อมูลบนเว็บ
  - ไม่ hallucinate ข้อมูลที่ไม่มี → ตอบว่า "ขอแนะนำให้ติดต่อ BOI โดยตรง"
- ภาษา: รองรับ 7 ภาษาเหมือนเว็บ
- ตัวอย่างคำถาม:
  - "What industries are eligible for BOI?" → ตอบจาก eligible activities data
  - "How to apply for Smart Visa?" → ตอบพร้อมลิงก์ไปหน้าที่เกี่ยวข้อง
  - "What tax incentives can I get in EEC?" → ตอบจาก incentives data
- **Safety**: มี content moderation, ไม่ตอบคำถามนอกขอบเขต BOI

**Script**:
> "เราเสนอฟีเจอร์เพิ่มเติมที่ไม่ได้อยู่ใน TOR แต่เราเชื่อว่าจะทำให้เว็บ BOI ก้าวนำหน้า — 'น้องอะไร' AI Chatbot ชื่อ Arai — ภาษาไทยแปลว่า 'อะไร' ต่างชาติก็ออกเสียงได้ง่าย ตรงประเด็น 'ถามอะไรก็ตอบ' น้องอะไรเชื่อมกับ OpenRouter ใช้ AI อย่าง Claude แต่ที่สำคัญ ข้อมูลที่ AI ตอบถูกควบคุมอยู่ — ตอบได้แค่ข้อมูลที่เรา train ไว้กับข้อมูลบนเว็บ BOI เท่านั้น ไม่มี hallucination ไม่มีข้อมูลผิด ถ้าไม่รู้ จะแนะนำให้ติดต่อ BOI โดยตรง"

**กรรมการอาจถาม**:
- "AI จะตอบข้อมูลผิดไหม?" → RAG architecture + guardrails + ถ้าไม่มั่นใจจะแนะนำติดต่อ BOI
- "ค่าใช้จ่าย AI เท่าไหร่ต่อเดือน?" → OpenRouter ใช้แบบ pay-per-use, ค่าใช้จ่ายต่ำ, สามารถ set budget cap ได้
- "PDPA กับ AI chatbot?" → ไม่เก็บข้อมูลส่วนตัว, conversation ไม่ถูก store ถ้าผู้ใช้ไม่ยินยอม
- "ถ้า OpenRouter ล่ม?" → Fallback to static FAQ, ไม่กระทบเว็บหลัก

## คำถาม Critical: "ทำไมต้องเลือก SYNERRY แทน ClickNext ที่มี 600 คน?"

**กลยุทธ์ตอบ**:
> "BOI ไม่ได้ต้องการ 600 คน — ต้องการ 16 คนที่ใช่

> 2. ทีม 16 คน ทุกคนเคยทำเว็บภาครัฐ ไม่ต้อง learn on the job
> 3. ส่งมอบตรงเวลา 100% — track record พิสูจน์
> 4. ราคาเหมาะสม — ไม่บวก overhead ของบริษัทใหญ่"

## คำถาม: "Winter ทำ Creative ดีกว่าไหม?"

> "Design สวยคือเรื่องพื้นฐาน — เว็บ BOI ต้อง:
> 1. Secure ผ่าน OWASP + Penetration Testing 2 ครั้ง

> Winter ทำ campaign สวยๆ ได้ แต่ทำเว็บภาครัฐมาตรฐาน .go.th ได้ไหม?"
